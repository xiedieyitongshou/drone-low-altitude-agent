# 数据库表结构草案

本文档基于当前实际代码结构更新，目标是支撑 `Day 18` 的历史任务复盘，同时也作为后续 ORM 建模和数据库落地的统一基准文档。当前系统已经形成三层数据流：

- Provider 原始响应层：和风天气返回的原始结构
- Mapper / Adapter 层：把原始响应转换成内部统一模型
- 内部统一模型层：规则引擎、推荐模块、比选模块稳定依赖的数据结构

数据库设计必须体现这三层边界。否则后续一旦更换天气供应商，历史数据、规则结果和复盘接口都会被牵连。

## 1. 设计原则

### 1.1 三层落库原则

- 原始快照可以保存，但只作为审计和排错用途
- 规则、推荐、比选只依赖内部统一模型对应的落库结构
- 业务结果表不直接依赖和风天气原始字段名

### 1.2 复盘优先原则

Day 18 的目标不是先做复杂分析，而是先保证以后可以回答这些问题：

- 当时传入了什么请求
- 当时查到了什么天气和预警
- 规则为什么给出这个结论
- 推荐窗口和多地点排序是怎么得出来的

### 1.3 供应商可替换原则

- 表结构不能和某一家天气 API 深度耦合
- 原始响应允许保留 `raw_payload_json`
- 真正给规则引擎和复盘页面消费的字段，必须来自统一 Schema

## 2. 为什么这里体现了 mapper 层的必要性

如果没有 mapper 层，数据库设计会直接出现两个问题：

- 天气字段和预警字段会完全跟着和风天气接口走
- 后续更换数据源时，表结构、规则引擎和历史复盘都要一起改

有了 mapper 层之后：

- 原始响应可以落到“原始快照表”
- 标准化后的天气、预警、评估结果落到“内部模型表”
- 规则引擎和复盘模块只读内部模型表

这就是数据库层面上 mapper 的价值：**把外部接口波动隔离在边界层之外**。

## 3. 建议的分层表结构

## 3.1 请求与任务主表

### `task_requests`

存储一次业务请求的原始输入，是所有后续快照和结果的主索引。

建议字段：

- `id`：主键
- `request_id`：业务请求 ID，唯一索引
- `request_type`：请求类型，如 `evaluate` / `recommend` / `compare`
- `location_text`：单地点原始输入
- `locations_json`：多地点原始输入列表
- `task_date`
- `start_time`
- `end_time`
- `task_type`
- `purpose`
- `scan_hours`
- `min_window_hours`
- `comparison_mode`
- `top_k`
- `raw_request_json`
- `created_at`

说明：

- 单地点请求使用 `location_text`
- 多地点比选请求使用 `locations_json`
- 保留 `raw_request_json` 便于完整复盘原始入参

## 3.2 地点标准化表

### `locations`

存储标准化地点信息，对应内部 `LocationInfo`。

建议字段：

- `id`
- `provider_name`：如 `qweather`
- `provider_location_id`
- `name`
- `latitude`
- `longitude`
- `adm1`
- `adm2`
- `country`
- `timezone`
- `utc_offset`
- `created_at`

建议索引：

- 唯一索引：`provider_name + provider_location_id`

说明：

- 不直接把 `provider_location_id` 叫成 `location_id`
- 这样以后换供应商时，语义更清楚

## 3.3 Provider 原始响应快照层

这一层是数据库里对应 provider/raw 的落库位置。

### `weather_provider_snapshots`

记录一次对天气供应商的查询动作和原始返回。

建议字段：

- `id`
- `request_id`
- `location_id`
- `provider_name`
- `snapshot_type`：`geo` / `hourly_weather` / `warning`
- `provider_endpoint`
- `provider_update_time`
- `raw_payload_json`
- `created_at`

说明：

- 这一张表只解决“当时第三方返回了什么”
- 不建议规则模块直接读取这里

## 3.4 内部统一天气模型表

这一层对应 mapper 之后的统一模型，是规则引擎真正依赖的数据。

### `weather_hourly_snapshots`

对应内部 `WeatherHourData`。

建议字段：

- `id`
- `request_id`
- `location_id`
- `provider_snapshot_id`
- `update_time`
- `fx_time`
- `temp`
- `text`
- `wind_scale`
- `wind_speed`
- `humidity`
- `precip`
- `pop`
- `pressure`
- `cloud`
- `created_at`

建议索引：

- `request_id + location_id + fx_time`

说明：

- 这是标准化后的逐小时天气表
- 规则引擎、推荐窗口、比选模块都应依赖它

### `weather_warning_snapshots`

对应内部 `WarningData`。

建议字段：

- `id`
- `request_id`
- `location_id`
- `provider_snapshot_id`
- `warning_id`
- `event_type`
- `warning_level`
- `title`
- `sender`
- `publish_time`
- `start_time`
- `end_time`
- `status`
- `text`
- `created_at`

说明：

- 如果无预警，则该请求下这一表允许没有记录
- 不需要人为插“空预警”假数据

## 3.5 规则评估结果层

### `cruise_assessments`

存储单地点评估的整体结果。

建议字段：

- `id`
- `request_id`
- `location_id`
- `allow_cruise`
- `overall_decision`
- `summary_risk_factors_json`
- `created_at`

说明：

- 对应当前 `CruiseAssessmentResponse.advice`
- 这是复盘时最常查的一层

### `cruise_hourly_assessments`

存储单地点逐小时规则结果。

建议字段：

- `id`
- `assessment_id`
- `fx_time`
- `decision`
- `risk_factors_json`
- `weather_snapshot_id`
- `created_at`

说明：

- 这里不建议直接存整段 `weather_json`
- 更合理的是通过 `weather_snapshot_id` 回连标准化天气表
- 这样更规范，也更方便后续查询

## 3.6 推荐结果层

### `recommendation_results`

存储一次推荐请求的整体输出。

建议字段：

- `id`
- `request_id`
- `location_id`
- `strategy_json`
- `total_candidates`
- `created_at`

### `recommendation_windows`

存储推荐窗口明细。

建议字段：

- `id`
- `recommendation_result_id`
- `rank`
- `start_time`
- `end_time`
- `duration_hours`
- `overall_decision`
- `risk_score`
- `reasons_json`
- `created_at`

说明：

- 推荐窗口是对规则结果的二次加工结果
- 单独拆表后，后续做窗口统计更方便

## 3.7 多地点比选结果层

### `comparison_results`

存储一次多地点比选请求的整体结果。

建议字段：

- `id`
- `request_id`
- `comparison_mode`
- `top_k`
- `recommended_location_id`
- `created_at`

### `comparison_location_rankings`

存储每个地点的排序结果和评分拆解。

建议字段：

- `id`
- `comparison_result_id`
- `location_id`
- `rank`
- `overall_decision`
- `allow_cruise`
- `risk_score`
- `score`
- `flyable_hour_count`
- `max_continuous_flyable_hours`
- `earliest_flyable_time`
- `window_quality_score`
- `score_breakdown_json`
- `summary_risk_factors_json`
- `best_window_json`
- `created_at`

说明：

- 这张表正好对应当前增强后的 `Day 17` 输出结构
- 后续如果改权重策略，只改比选逻辑，不必改主表结构

## 3.8 反馈与复盘扩展层

### `task_feedbacks`

为后续反馈学习和真实任务回写预留。

建议字段：

- `id`
- `request_id`
- `assessment_id`
- `comparison_result_id`
- `recommendation_result_id`
- `was_executed`
- `actual_result`
- `user_feedback`
- `comment`
- `created_at`

## 4. Day 18 最小可用落库范围

如果先做最小闭环，不建议一次把所有表都建满。最小可用版本建议先建这几张：

- `task_requests`
- `locations`
- `weather_provider_snapshots`
- `weather_hourly_snapshots`
- `weather_warning_snapshots`
- `cruise_assessments`
- `cruise_hourly_assessments`

这样已经足够支撑：

- 单地点历史复盘
- 请求追踪
- 天气与预警快照回看
- 规则结论回放

如果要把第三周功能一起纳入复盘，再继续补：

- `recommendation_results`
- `recommendation_windows`
- `comparison_results`
- `comparison_location_rankings`

## 4.1 Day 18 建表顺序建议

建议实际落地时按这个顺序建：

1. `task_requests`
2. `locations`
3. `weather_provider_snapshots`
4. `weather_hourly_snapshots`
5. `weather_warning_snapshots`
6. `cruise_assessments`
7. `cruise_hourly_assessments`

原因：

- 先有请求主表
- 再有地点与天气快照
- 最后落评估结果表

## 4.2 Day 18 最小闭环验收标准

如果以下能力都能跑通，就可以认为 `Day 18` 最小闭环完成：

- `/cruise/evaluate` 评估完成后自动写入上述核心表
- 可以通过 `request_id` 查回一条历史任务
- 可以看到：
  - 当时输入
  - 标准化天气
  - 标准化预警
  - 整体评估结果
  - 逐小时评估结果

## 5. 与当前代码结构的对应关系

### Provider/raw 层

- `app/services/weather/schemas.py`
- `app/services/weather/service.py`

- `weather_provider_snapshots`

- `app/services/weather/mapper.py`
- `app/services/weather/warning_extractor.py`

职责：

- 这一层通常不单独建业务表

- `cruise_assessments`
- `cruise_hourly_assessments`

## 6. 现阶段数据库推荐

结合当前开发阶段：

- 本地最小实现：`SQLite + SQLAlchemy`
- 后续部署正式版：`PostgreSQL + SQLAlchemy`

原因：

- 当前先要把 Day 18 的复盘闭环跑通
- 后续如果做更复杂查询、并发和 RAG/向量能力，再迁到 PostgreSQL 更合理
