# 数据库表结构草案

本文档用于支撑 `Day 4` 的数据结构规划，目标是让后续天气查询、规则判断、历史任务复盘和用户反馈都有明确落库位置。

## 1. 设计原则

- 主链路先围绕“任务请求 -> 天气快照 -> 预警快照 -> 评估结果”设计
- 结构尽量和 API 输出模型保持一致，降低映射复杂度
- 先满足单地点评估，后续再扩展多地点比选和推荐窗口

## 2. 建议表结构

### 2.1 `task_requests`

存储一次任务评估请求的原始输入。

核心字段：

- `id`：主键
- `request_id`：业务请求 ID
- `location_text`：用户输入地点
- `task_date`：任务日期
- `start_time`：开始时间
- `end_time`：结束时间
- `task_type`：任务类型
- `purpose`：任务说明
- `created_at`：创建时间

### 2.2 `locations`

存储地点解析后的标准化结果，便于复用和缓存。

核心字段：

- `id`：主键
- `location_id`：和风天气地点 ID
- `name`
- `latitude`
- `longitude`
- `adm1`
- `adm2`
- `country`
- `timezone`
- `utc_offset`
- `created_at`

### 2.3 `weather_hourly_snapshots`

存储一次查询得到的逐小时天气快照。

核心字段：

- `id`
- `request_id`
- `location_id`
- `update_time`
- `fx_time`
- `temp`
- `text`
- `wind_scale`
- `wind_speed`
- `humidity`
- `precip`
- `pop`
- `pressure`
- `cloud`
- `created_at`

说明：

- 一次任务请求会对应多条逐小时天气记录

### 2.4 `weather_warning_snapshots`

存储天气预警快照。

核心字段：

- `id`
- `request_id`
- `location_id`
- `warning_id`
- `event_type`
- `warning_level`
- `title`
- `sender`
- `publish_time`
- `start_time`
- `end_time`
- `status`
- `text`
- `created_at`

### 2.5 `cruise_assessments`

存储一次任务评估的整体输出。

核心字段：

- `id`
- `request_id`
- `allow_cruise`
- `overall_decision`
- `summary_risk_factors_json`
- `created_at`

### 2.6 `cruise_hourly_assessments`

存储逐小时评估结果。

核心字段：

- `id`
- `assessment_id`
- `fx_time`
- `decision`
- `risk_factors_json`
- `weather_json`
- `created_at`

### 2.7 `task_feedbacks`

为后续反馈学习和历史复盘预留。

核心字段：

- `id`
- `request_id`
- `assessment_id`
- `user_feedback`
- `was_executed`
- `actual_result`
- `comment`
- `created_at`

## 3. 第一阶段最少落库范围

如果前期只做最小闭环，建议优先落这四张表：

- `task_requests`
- `weather_hourly_snapshots`
- `weather_warning_snapshots`
- `cruise_assessments`

这样已经足够支持：

- 查询复盘
- 结果追踪
- Demo 展示

## 4. 与当前 Schema 的对应关系

- 请求参数：`app/schemas/request.py`
- 天气数据：`app/schemas/weather.py`
- 预警数据：`app/schemas/warning.py`
- 评估输出：`app/schemas/assessment.py`

后续 ORM 设计时，建议数据库字段名尽量贴近这层统一 Schema。
