# 低空作业气象决策 Agent 开发计划与时间表

## 1. 计划说明

本计划基于 [项目重构技术方案](D:\desktop\YYagent\项目重构技术方案.md) 制定，目标是将当前原型重构为一个可本地开发、可服务器部署、可上传 GitHub 的低空作业气象决策 Agent 求职项目。

时间假设：
- 每天可投入 `4-5 小时`
- 以 `5 周` 为一个完整开发周期
- 目标不是一次性堆满技术点，而是每周都产出可运行、可展示、可写进简历的阶段成果

开发优先级：
1. 先做核心主链路
2. 再做功能增强
3. 再做 Agent 能力
4. 最后做 RAG、部署和包装

三者怎么配合
你写 FastAPI 应用
用 Pydantic 定义请求/响应/配置模型
用 Uvicorn 启动服务
可以理解成：
FastAPI = 框架
Pydantic = 数据规则
Uvicorn = 运行服务器

## 4. 后续延后实现内容

以下内容不影响当前主链路闭环，但从系统完整性看是明确的后续增强项。其中 4.1 和 4.2 已完成接口预留，暂未接入实际大模型 API。

### 4.1 自然语言解析升级（已完成接口预留）
- 当前版本：`规则 + 关键词 + 正则 + dateparser`
- 后续版本：在 Day 31 与 LLM 解释层一起引入大模型结构化解析
- 建议原则：保留当前输入输出结构，新增大模型解析器，Day 22 规则解析器作为 fallback
- 推荐依赖：`openai` + `pydantic`，可选 `instructor`

### 4.2 大模型统一接入方案（已完成接口预留）
- 目标：把自然语言解析升级和 LLM 解释层放到同一天处理
- 统一入口：新增 `llm_client.py`，避免自然语言解析和解释生成各接一套模型调用
- 推荐能力拆分：
  - `parse_task_request(...)`：负责自然语言到结构化参数
  - `generate_final_response(...)`：负责基于规则结果、RAG 片段、建议条目生成用户可读说明
- 推荐依赖：
  - 必选：`openai`、`pydantic`
  - 可选：`instructor`、`tenacity`
- 设计原则：
  - Day 22 规则解析器继续保留为 fallback
  - 大模型只负责结构化解析和表达生成，不替代规则引擎判断
  - 规则结果仍然是最终结论来源

### 4.3 RAG 检索升级方案
- 当前版本：`scikit-learn TF-IDF + cosine similarity`
- 当前定位：最低可运行 RAG 检索原型，适合先跑通知识库、索引、检索、建议返回流程
- 后续推荐方案一：`OpenAI Embeddings + FAISS`
  - 优点：真正语义检索、本地向量索引、工程量可控、适合求职项目展示
  - 推荐依赖：`openai`、`faiss-cpu`、`numpy`
  - 适合阶段：当前知识库规模较小、暂时不想引入数据库级向量存储时
- 后续推荐方案二：`OpenAI Embeddings + Chroma`
  - 优点：更接近现成向量库，支持 collection、metadata、持久化管理
  - 推荐依赖：`openai`、`chromadb`
  - 适合阶段：希望展示完整向量库能力，并且能接受更重依赖时
- 建议原则：
  - 先保持当前 TF-IDF 版本跑通完整链路
  - 等第五周全部功能稳定后，再把 `vector_knowledge_store.py` 替换为 embedding + 向量库实现
  - 保持 `/knowledge/advice/retrieve` 接口不变，减少对上层业务的影响

### 4.4 Session Memory 升级
- 当前版本：`TTLCache` 进程内缓存
- 后续版本：Redis 版会话记忆
- 建议原则：保持 `session_memory.py` 抽象层不变，只替换底层存储实现

### 4.5 Profile Memory 与用户体系
- 当前阶段：延后
- 后续版本：与 `user_id`、角色、组织、权限、长期偏好一起设计
- 建议原则：不要把长期用户体系和当前会话记忆混成一层

### 4.6 对话内容与用户偏好持久化
- 后续可增加对话记录表
- 后续可增加用户偏好表
- 后续可支持历史偏好学习、默认参数补全、输出风格偏好

## 6. 建议的执行策略

### 6.1 每天开发顺序
建议每天按以下节奏：

1. 前 30 分钟  
   回顾昨天进度，明确今天唯一主目标

2. 中间 3 到 3.5 小时  
   只做一个模块，不要多线程切换

3. 最后 30 到 60 分钟  
   测试、提交代码、写当天记录

### 6.2 每周固定动作
每周最后一天做三件事：
- 清理代码
- 补文档
- 记录下周要做的第一件事

### 6.3 对后续增强项的处理原则
对于以下内容，不要在当前主链路尚未稳定时提前做重型实现：
- 大模型自然语言解析与 LLM 解释层
- RAG 从 TF-IDF 升级为 embedding + 向量库
- Redis 会话记忆替换
- 完整用户体系与 Profile Memory
- 全量对话持久化

当前更合理的策略是：
- 先把可运行原型做出来
- 再把易变层做成可替换模块
- 最后再做正式工程化增强

### 6.4 开发顺序不要打乱
不要一开始就上：
- 强化学习
- 复杂多 Agent
- 过度前端美化
- 完整用户权限体系

正确顺序是：
- 核心链路
- 规则引擎
- 功能增强
- Agent 编排
- Session Memory
- 长期用户体系
- RAG
- 反馈优化

## 8. 一句话执行建议

这次重构最重要的不是“技术点尽量多”，而是：

`每周都交付一个能运行、能展示、能写进简历的阶段成果。`

当前应该优先保证：
- 主链路稳定
- 模块边界清晰
- 易变层可替换

在这个基础上，再逐步补：
- 大模型自然语言解析与 LLM 解释层
- embedding + FAISS / Chroma 语义检索
- Redis 会话记忆
- 用户体系与长期偏好
- RAG 与部署
- 前端展示控制台

只要按这个节奏推进，后端 5 周内完成核心闭环，第 6 到第 7 周补齐前端展示后，它就不再是“比赛作品重写”，而是一套完整的低空作业气象决策 Agent 求职项目。
