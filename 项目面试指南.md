## 七、Day 3 为什么要这样设计

### 7.1 为什么要单独封装天气服务层

- 不把第三方接口调用散落在业务代码里
- 更换供应商时改动集中
- 更方便测试、监控和重构

### 7.2 为什么拆成 `service + client + schema + exceptions`

- `service` 负责业务语义
- `client` 负责请求细节
- `schema` 负责响应建模
- `exceptions` 负责错误分类

### 7.3 为什么要统一处理超时、重试和异常

- 第三方 API 是天然不稳定外部依赖
- 如果不收口，下游规则引擎会被网络失败和脏响应污染

## 十、Day 4 解决了什么问题

### 10.1 不只是“转格式”

这部分不是简单把 API 数据改个字段名，而是统一系统的数据契约。

它主要做三件事：

- 定义输入结构
- 定义中间传输结构
- 定义输出结构

### 10.2 统一了哪些数据

当前已经统一：

- 请求参数
- 地点信息

### 10.3 为什么这很重要

- 避免规则引擎直接耦合第三方字段名
- 更换天气供应商时改动范围更小
- API 层、服务层、规则层之间字段更一致

## 十一、Day 4 为什么要这样设计

### 11.1 为什么把第三方响应模型和内部统一模型分开

当前项目有两层模型：

- `app/services/weather/schemas.py`
- `app/schemas/*.py`

原因：

- 第一层更贴近外部天气 API 响应
- 第二层更贴近项目内部主链路数据

### 11.2 为什么请求、天气、预警、输出要分别建模

- 每一类数据承担的业务职责不同
- 独立建模后更容易复用、校验和测试

### 11.3 为什么要同时规划数据库表结构

- 统一 Schema 不只为接口传输，也要为后续落库提供稳定对应关系

数据库草案文件：

- `docs/数据库表结构草案.md:1`

## 十五、Day 5 的关键逻辑怎么讲

### 15.1 日期格式校验

文件：`app/schemas/request.py:22`

讲法：

- 用标准日期解析来保证 `YYYY-MM-DD` 格式合法
- 不是只看字符串长度，而是校验它是不是有效日期

### 15.2 时间格式校验

文件：`app/schemas/request.py:31`

讲法：

- 允许标准 `HH:MM`
- 同时兼容 `24:00` 作为结束时间
- 明确禁止 `24:00` 作为开始时间

### 15.3 跨天时间段处理

文件：`app/schemas/request.py:42`

讲法：

- 如果 `end_time <= start_time`，自动视为跨天
- 生成标准化后的开始、结束时间戳
- 用 `spans_next_day` 明确标记这个时间段是否跨天

### 15.4 统一错误响应

文件：`main.py:43`

讲法：

- 任何请求校验错误都返回同一套结构
- 前端或调用方不需要针对不同字段错误写很多分散逻辑

### 15.5 为什么要先做输入校验预览接口

文件：`main.py:89`

讲法：

- 当前规则引擎还没接进来，所以我先用这个接口把“输入合法性”和“标准化结果”单独跑通
- 这样后面接规则引擎时，输入边界已经稳定

## 十六、Day 5 为什么要这样设计

### 16.1 为什么要把校验前置

- 非法输入不能进入天气查询和规则判断
- 否则会浪费外部 API 调用次数，也会放大后续异常处理复杂度

### 16.2 为什么跨天逻辑要放在请求模型里

- 跨天本质上属于输入解释的一部分
- 提前标准化后，后续天气提取和规则判断只消费标准结果

### 16.3 为什么定义统一错误响应

- 调用方需要稳定错误结构
- 这样前端、日志和监控都更容易处理

### 16.4 为什么要加接口级异常处理

- 框架默认错误虽然能返回信息，但不一定符合项目自己的错误契约
- 项目化开发需要统一错误输出风格

## 二十、一段可以直接背的综合表达

可以直接这样讲：

> 在这个项目里，我先把和风天气能力封装成独立服务层，用 `service.py` 提供地点解析、逐小时天气和天气预警三个业务接口，用 `client.py` 统一处理 HTTP 请求、超时、重试和异常分类。然后我把请求参数、天气数据、预警数据和评估结果统一建模，形成系统内部稳定的数据契约。接着我又补了输入校验层，负责日期时间校验、跨天时间段标准化和统一错误响应。这样做的核心目的，是把第三方 API 的不确定性、输入的不确定性都限制在系统边界，让后续规则引擎和 API 层只依赖内部标准结构。
